# 记忆系统哲学 — 人脑启示 vs AI 设计

> 日期: 2026-05-20
> 关联: `tools/memory_tool.py`, `agent_core/memory_extractor.py`, `agent_core/context_builder.py`, `agent_core/memory_manager.py`, `tools/registry.json`
> 源对话: 坐山客 Agent 与用户张清泉关于记忆系统 v2 改造后的深入讨论

---

## 一、人的记忆是什么样的

### 1.1 记忆不是数据库

最核心的区别。人的记忆不是「存了就能读回来」——每次回忆都是一次**重构**，不是**回放**。

你说「昨天吃了啥」，大脑不是打开一个文件读取昨天的记录，而是根据当前场景、情绪、关联线索，**现场拼凑**出一段叙事。所以：
- 同一个回忆，不同时间想起来细节不一样
- 容易受暗示——「你昨天是不是吃了火锅？」可能会让你记错了

**对 AI 的启示**：记忆块注入时带「仅供参考，不相关可忽略」。AI 的记忆也应该是**参考而非真理**。

### 1.2 记忆分三层（Atkinson-Shiffrin 模型）

| 层次 | 容量 | 持续时间 | 对应 AI 概念 |
|------|------|---------|-------------|
| **感觉记忆** (Sensory) | 极大 | 毫秒级 | 当前输入 token 流 |
| **短期/工作记忆** (Short-term) | 7±2 个 chunk | 秒~分钟 | 当前对话上下文 (Agent Loop 中的 `messages`) |
| **长期记忆** (Long-term) | 几乎无限 | 天~年 | MEMORY.md / AgentMemory 表 |

**健忘的核心原因之一**：短期记忆容量有限。正想着 A 事突然被打断去想 B 事，A 就被挤出去了。这就是「断片」的本质——不是 A 忘了，是 **A 从来没机会进入长期记忆**。

**对 AI 的启示**：
- Context window 就是 AI 的「短期记忆」。为什么需要 Agent Loop？因为工具调用的结果必须显式 append 到 `messages`（写回短期记忆），否则下一轮就不记得了。和「大脑走神就忘了刚才要干嘛」是一个道理。
- 超过 context window 的历史需要压缩/摘要，相当于人类的**信息压缩整合**。

### 1.3 长期记忆的细分

| 类型 | 定义 | 例子 | AI 对应 |
|------|------|------|---------|
| **陈述性记忆** | 可以「说出来」的知识 | 用户叫张清泉 / 柏林是首都 | `AgentMemory` 表中的条目 |
| └ 语义记忆 | 事实 | 冷色系看起来酷 | memory 工具存的内容 |
| └ 情景记忆 | 个人经历 | 昨天改了一下午代码 | `session_search` 搜索聊天记录 |
| **程序性记忆** | 怎么做（无法言说） | 打字不需要想每个字母的位置 | LLM 参数权重（训练好的模型参数） |

**为什么这个区分重要**：
- 我们的 `memory` 工具只存了**语义记忆**。
- 人真正的价值在于**程序性记忆**（例如骑自行车不需要回想步骤）。
- AI 的「程序性记忆」是训练好的模型参数，不是我们能通过 memory 工具管理的。

## 二、断片（Tip-of-the-Tongue）

### 2.1 本质

**记忆检索失败**，不是存储失败。

证据：
1. 你「知道」这个人的名字——有**熟悉感**
2. 你能想起第一个字母、音节数、相关的场景
3. 但就是**说不上来**——检索通路堵了
4. 过一会儿突然想起来了——这说明它一直在那，只是检索路径暂时不通

### 2.2 原因

人的记忆是**联想网络**。一个记忆节点通过多条路径连接到其他概念。断片是因为当前激活模式没有覆盖到那个节点——**线索不够**。

### 2.3 AI 对应

我们的 `_build_memory_block()` 用了 topic 匹配 + 权重排序。如果当前话题是「电影」但用户关于电影的记忆权重较低或 topic 不匹配，就可能注不进上下文。LLM 就会「想不起来」。

解决方式：P0 级别的「始终注入」通道——相当于人脑的「熟悉感」标记。

## 三、健忘症（Amnesia）

| 类型 | 问题 | 示例 | AI 对应 |
|------|------|------|---------|
| **顺行性遗忘** (Anterograde) | 无法形成**新**记忆 | 看了 10 遍报纸，每一遍都觉得是新的 | 如果 LLM 不调 `memory(add)`，信息留在当前 session 内，下个 session 完全不记得 |
| **逆行性遗忘** (Retrograde) | 无法回忆起**旧**记忆 | 记得今天吃了什么，但想不起来去年的事 | `session_search` 检索失败，或记忆 weight 衰减到 0 |

**顺行性遗忘的根因**：海马体受损 → 短期记忆无法 consolidate 到长期记忆。

**AI 对应**：LLM 不调 `memory(action='add')`，信息就在当前 context 里蒸发了。这就是 AI 的「顺行性遗忘」。

而旧版 **MemoryExtractor 快速通道** 相当于试图做一个自动的海马体——对话结束强行提取记忆。但它精度不够。

## 四、对坐山客记忆系统的哲学检验

### 4.1 「忘」是功能还是 bug？

人的大脑会主动**忘掉**大部分事情，只保留重要的：
- 忘记今天早餐吃什么 → 正常，不重要
- 忘记自己叫什么 → 病态

**我们旧版的问题**：什么都想记。快速通道见「记住」就存，LLM 提取每次对话都跑。结果记忆池里堆满了低置信度的半成品。

**新版设计**：只有 LLM 自主决定存的才进长期记忆——相当于人类海马体的选择性巩固 gate。权重衰减机制（P3 自动清理）模拟自然的遗忘。

### 4.2 记忆应该「精确」还是「模糊」？

人类记忆是**模糊而稳定的**——记得「上海比北京暖和」，但不太可能记得「2023年8月15日下午3点上海气温32.5°C」。

`memory` 工具存的是**陈述句**（语义记忆），不是精确数字。需要精确值时让 LLM 调工具（`get_weather`），不需要记在记忆里。

### 4.3 「自己管理」还是「后台自动」？

| 模式 | 类比 | 优点 | 缺点 |
|------|------|------|------|
| **后台自动提取**（旧版） | 给健忘症患者配自动记录仪 | 不依赖 LLM 判断力 | 什么都记，质量差，重复多 |
| **LLM 自主管理**（新版） | 让人自己决定哪些事记在本子上 | 选择性更强，质量更高 | 依赖 LLM 判断力，可能漏记重要的 |

人脑还有一个**情感触发机制**：情绪强度高（惊吓、惊喜、恐惧）的事件自动进入长期记忆，不需要主动「记住」。AI 目前缺少这个——LLM 需要主动调 tool。未来可以加「情感阈值」——LLM 回复中「这很重要」「关键」「核心」措辞 → 自动触发 `memory(add)` 建议。

## 五、下一步改进方向

### 5.1 让 LLM 学会主动用 memory 工具（当前最大瓶颈）

这是改造后最关键的 gap。系统 prompt 已经加了「记忆能力」说明，但 LLM 会不会主动调 `memory(add/read/replace/remove)` 还不一定。

**改进措施**：
- 在 `memory_tool` 的 description 中强化使用引导
- 在系统 prompt 的记忆能力节加一句「当用户告诉你关于他自己的新信息时，考虑用 memory 存一下」
- **最关键的一个设计问题**：是否有必要在第一次对话结束时或在后端检测到新信息时，自动给予 LLM 一个内存建议？或者在初始提示中更明确地指导 LLM 的习惯形成？

### 5.2 情景记忆的缺失

当前只支持**语义记忆**（事实型）。没有**情景记忆**（"上次我们讨论数据库schema的时候……"）。

情景记忆其实由 `session_search` 承担——搜索历史聊天记录。但需要确保 LLM 在需要回溯时主动调 `session_search`（已修复意图混淆陷阱）。

### 5.3 情感/重要性触发的自动巩固

人脑有自动的门控机制：情感强度高的事件→自动 consolidate。

我们系统缺一个类似的**自动触发**——当 LLM 回复中出现「很重要」「关键」「核心」「务必记住」等标记时，自动回调 `memory(add)` 或产生建议。这比全自动提取更可控。

### 5.4 记忆的分层展示与治理

当前记忆只有权重+等级（P0-P3），但缺乏：
- 记忆的来源追溯（哪次对话产生的？）
- 记忆的有效性标记（用户确认过？还是 AI 自己推测的？）
- 记忆的版本历史（修改过几次？）
- 记忆间的关联（哪些记忆是同一主题的）

### 5.5 P0/P1 的自适应阈值

`P0_THRESHOLD = 8.0` 等常量是硬编码的。随着记忆积累，权重分布会变化。更好的方式是：
- 动态阈值（按记忆总数百分位计算）
- 或者记忆池上限管理（如同 Hermes 的 2200/1375 字符限制）

### 5.6 程序性记忆的缺失

这个目前无法用 memory 工具实现——程序性记忆是模型训练参数。但可以**变相实现**：
- 将「在什么场景下用什么 workf」的经验写成 skill（SKILL.md），通过 skill 匹配注入
- 坐山客已经支持 skill 的 `_build_skill_block()` 注入

## 附录：人类记忆与 AI 记忆对照表

| 人的记忆特性 | 当前 AI 对应 | 差距 |
|-------------|-------------|------|
| 记忆是重构不是回放 | 记忆块标记「仅供参考」 | ✅ 已实现 |
| 短期记忆容量有限 | Agent Loop messages 上限 + context window | ⚠️ 压缩策略待优化 |
| 需要注意力才能进长期 | LLM 主动调 memory 工具 | ⚠️ LLM 使用习惯待培养 |
| 检索失败 = 断片 | topic 不匹配导致不注入 | ✅ P0 始终注入兜底 |
| 海马体选择性巩固 | 无自动提取，全靠 LLM 判断 | ⬆️ 下一步改进方向 |
| 情感触发自动巩固 | 无 | ⬆️ 未来可加 |
| 程序性记忆 (技能) | 模型参数 + skill 注入 | ✅ skill 系统已存在 |
| 情景记忆 (过往经历) | session_search 搜索历史 | ✅ 已实现 |
| 遗忘 = 功能 | 权重衰减 + P3 自动清理 | ✅ 已实现 |
| 联想式检索 | Topic 匹配 + 权重排序 | ✅ 已实现 |
