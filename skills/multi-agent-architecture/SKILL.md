# 多Agent协作架构设计

## 1. 单 Agent vs 多 Agent 决策框架

### 四维测试

当**多个**条件成立时，多 Agent 是值得的：

| 维度 | 信号 |
|------|------|
| **错误成本** | 高 + 发现晚（法律、医疗、金融、芯片验证） |
| **知识领域** | 不可简化到一个领域（跨学科、跨法域） |
| **多方签署** | 输出需要不同视角的审批 |
| **决策不可逆** | 出错后难以低成本重做 |

### 核心原则

多 Agent = 「用协作开销换取容错能力」。如果任务没什么可错的，协作开销就是净损失。

### 两个前提（硬要求）

多 Agent 只有在两个条件都满足时才优于单 Agent：
1. **共享关键上下文** — 所有 Agent 访问相同的真相源
2. **实时进度同步** — Agent 之间语义级别通信

## 2. 冲突仲裁

### 核心问题

**LLM 做不了仲裁者。** 原因：
- 谄媚 — 倾向于同意读到的最新内容
- 近因偏差 — 偏爱最后读到的内容
- 确认偏差 — 验证而非质疑
- 核心能力是补全，不是判断

### 四种仲裁方法

#### 方法一：溯源差异（Traceability Diff）
- 调度者不「判断」，只检测分歧
- 每个 Agent 输出结构化，附带 `requirement_ref`
- 同一 `requirement_ref` + 不同结论 → 标记为冲突
- 调度者从「法官」降级为「差异检测器」

#### 方法二：对抗审查（Red Team）
- 独立 Red Team Agent：「只攻击，不批准」
- 流程：输出 → Red Team 攻击 → 严重性分级 → 摘要 → 人类决策
- 必须搭配引证要求 — Red Team 必须引用具体输出段落

#### 方法三：结构化输出 + 规则引擎
三层规则层次：
- **第一层**：领域宪法（人类预写）→ 可枚举、可靠、机器可执行
- **第二层**：可量化需求条款（自动提取）→ 有限、可靠
- **第三层**：需求检查清单（Agent生成）→ 不可机器执行，仅供参考

#### 方法四：升级（人类从菜单选择）
- Agent 把冲突格式化为成本/收益菜单
- 人类选择选项 — 不需要完整领域上下文

### 知识冲突 vs 价值冲突

**知识冲突** — 例：「密码最小长度是8还是6？」→ 可验证 → 可解决
**价值冲突** — 例：「PM说要实时，开发说只能轮询」→ 不可验证，双方都对，只是优化不同目标 → 需要人类价值判断

## 3. 成本量化框架

当价值冲突升级到人类时，Agent 系统能做的最好的事是把「冲突文本」转化为「成本表」：

```
冲突：需求 §7.1 "物流状态实时"

选项 A（PM立场）：WebSocket 实时推送
  成本：消息队列 + WebSocket 服务 + 3周
  风险：重构订单服务，影响7个消费者
  收益：预估 +2% 转化率

选项 B（开发立场）：5分钟轮询
  成本：最长5分钟延迟
  风险：无
  收益：零额外开发成本

▶ 决定：A 还是 B？
```

## 4. 仲裁容量分层

```
第一层：规则引擎自动裁决           → ~30%（可量化的冲突）
第二层：成本量化 + 格式化菜单      → ~40%（人类比较而非调研）
第三层：历史案例匹配               → ~20%（参考，非替代）
第四层：纯人类判断                 → ~10%（不可自动化）
```

## 5. 核心原则

> **任何不建立在人类预写规则之上的仲裁，最终都会退化为人类判断。**

多 Agent 系统的天花板不是取代人类价值判断，而是：
1. 把冲突从散文转化为结构化成本表
2. 让人类从「调研成本」变为「比较成本」
3. 用历史案例作为参考
4. 把真正困难的10%干净地交给人类

## 6. 聊天-任务并行会话

### 问题

当前 Agent 框架把每次交互视为单一会话和单一模型。这导致：
1. **模型在会话开始时就固定** — 闲聊也慢，复杂任务也快不了
2. **聊天和任务争夺同一上下文窗口** — 互相污染

### 解决方案：会话状态机

```
                    ┌───────────┐
                    │ 聊天模式   │
                    │ （轻模型） │
                    └─────┬─────┘
                          │ "动手吧"
                          ▼
                    ┌───────────┐
                    │ 任务模式   │
                    │ （重模型） │
                    └───────────┘
```

**核心原则：**
- 会话是群组，不是单数 — 一个用户 = 一个会话群组，多个并行会话
- 触发检测简单 — 「开始干活/帮我做/动手吧」→ 切到 TASK；「对了…/你说那个…」→ 开并行 CHAT
- 追溯只在 TASK 会话中存活 — 不污染聊天

## 7. 思维图 vs 行动图

### 双图框架

对 AI 伙伴的信任来自「可见的思考」，而不仅仅是「可见的行动」：

| | 思维图 (Thinking Map) | 行动图 (Action Map) |
|---|---|---|
| 回答的问题 | "我们在构建什么？" | "我们怎么构建？" |
| 方向 | 中心→向外（无流向） | START→END（有方向） |
| 节点 | 概念、领域、问题、约束、未知 | 动作、决策、结果 |
| 边 | 从属关系 | 因果/决策流 |
| 变更含义 | "我的理解变了" | "我的方案变了" |
| 用户编辑 | 自由编辑（拖拽+聊天） | 只读（暂停/停止） |

### 独立性与关联

两个独立的数据结构，不是同一数据的两视图。由「可执行叶子节点」链接：思维图中的可执行叶子 → 生成行动图块。每个行动图携带对源思维节点的反向引用。

### 状态锁

任何行动图处于 Running → 思维图 READ-ONLY。通过暂停、暂停-讨论、停止-编辑来解锁。

## 8. 行动图执行引擎

### 单节点执行器（关键边界）

**坐山客 Agent Loop 被当作单节点执行器，不是整图执行器。** 如果用坐山客执行整个行动图，跟踪可见性会塌缩为黑盒转圈——这正是双图架构要解决的问题。

| 坐山客能做 | 坐山客不能做 |
|-----------|-------------|
| 接收一个清晰的原子任务 | 遍历多个节点 |
| 调用工具执行 | 决定「下一步走哪条分支」 |
| 返回：成功/失败 + 结果摘要 | 判断「这个做对了吗？」 |
| | 自主决定重新规划 |

### 逐节点执行循环

```
exec 节点：
  1. 提取 verification_checks
  2. 运行前置检查（URL可达性、命令存在、文件存在）
  3. 前置检查失败 → 重试 / 跳转 fallback_node / 升级给用户
  4. 前置检查通过 → 构建单任务 prompt
  5. 执行 → 用户看到节点状态：运行中 ⏳
  6. 等待（超时可感知、重试可感知、失败可感知）
  7. 写入 result_summary + artifacts
  8. 如需审批 → 暂停，问用户
  9. 继续下一个节点

decision 节点：
  1. 匹配上游结果与规则（程序化，非 LLM）
  2. 首条匹配规则胜出
  3. 无匹配 → 回退（ask_user 或默认目标）

milestone 节点：
  1. 暂停，总结已完成节点
  2. 问用户：继续 / 暂停思考 / 停止并编辑地图
```

### exec 节点验证（非独立节点）

验证是 exec 节点的内置生命周期阶段，不是行动图上的独立节点（那会使节点数翻倍）。

### 动态回退

验证或执行失败时，可以「动态追加回退节点」到行动图。失败的路径保持可见（带 ❌ 状态）— 这是有价值的追溯数据。

### 设计原则

> **上层决定下层。** 数据结构跟随可视化需求，不是反过来。
