# 系统化调试

## 核心理念

**不在修复之前先做根因调查，不算真正的调试。**

症状修复就是失败。

## 铁律

```
无根因调查，不修复。
```

如果还没完成第一阶段调查，不能提修复方案。

## 四个阶段

必须完成每个阶段才能进入下一阶段。

---

## 第一阶段：根因调查

**在尝试任何修复之前：**

### 1. 仔细读错误信息

- 不要跳过错误或警告
- 它们常常包含精确的解决方案
- 完整阅读堆栈跟踪
- 注意行号、文件路径、错误码

### 2. 稳定重现

- 能可靠触发吗？
- 精确步骤是什么？
- 每次都发生吗？
- 如果不可重现 → 收集更多数据，不要猜测

### 3. 检查最近的变更

**首先运行 `git status`。** 未提交的改更是"之前好的现在坏了"的头号原因。

```bash
git status
git log --oneline -10
git diff
```

**当用户说某样东西"不见了"或"之前是好的"：** 检查 `git diff` 具体文件。如果 git diff 显示没碰那段代码，答案就是"它从来就不在那里"，而不是"让我继续读文件找问题"。

### 4. 跟踪数据流

当错误在调用栈深处时：
- 坏值从哪来的？
- 谁用坏值调了这个函数？
- 一路回溯直到找到源头
- 修复源头，不是症状

### 第一阶段完成清单

- [ ] 错误信息已完整阅读并理解
- [ ] 问题已稳定重现
- [ ] 最近变更已确认并审查
- [ ] 证据已收集（日志、状态、数据流）
- [ ] 问题已定位到具体组件/代码
- [ ] 根因假设已形成

**停：** 在不理解为什么会发生之前，不要进入第二阶段。

---

## 第二阶段：模式分析

### 1. 找工作中的示例

- 在同项目中找相似的工作代码
- 什么跟坏了的代码相似但能工作？

### 2. 对照参考

- 如果在实现某种模式，完整阅读参考实现
- 不要略读——读每一行
- 完全理解模式后再应用

### 3. 找出差异

- 工作版本和坏版本有什么区别？
- 列出每个差异，不管多小
- 不要假设"那个没关系"

---

## 第三阶段：假设与验证

### 1. 形成单一假设

- 明确陈述："我认为X是根因，因为Y"
- 写下来
- 具体，不要模糊

### 2. 最小化验证

- 做**最小**的改动来验证假设
- 一次只变一个变量
- 不要同时改多个东西

### 3. 验证后再继续

- 好了？ → 第四阶段
- 没好？ → 形成新假设
- 不要在上面叠更多修复

### 4. 当你不知道时

- 说"我不理解X"
- 不要假装知道
- 向用户寻求帮助
- 做更多调研

---

## 第四阶段：实施修复

### 1. 创建失败的测试用例

- 最简单的重现
- 修复前必须有

### 2. 实施单个修复

- 修复已确定的根因
- **一次只改一个**
- 没有"顺带"改进
- 没有捆绑重构

### 3. 验证修复

```bash
pytest tests/test_module.py::test_regression -v
pytest tests/ -q
```

### 4. 如果修复不奏效——三次规则

- **停**
- 数一下：尝试了几个修复？
- 如果 < 3：回第一阶段，用新信息重新分析
- **如果 ≥ 3：停，质疑架构**
- 不要在没有架构讨论的情况下尝试第4次修复

### 5. 如果3+次修复失败：质疑架构

**表明架构问题的模式：**
- 每次修复都在不同位置暴露新的共享状态/耦合
- 修复需要"大规模重构"才能实现
- 每次修复都在其他地方产生新症状

**停，质疑根本问题，与用户讨论。**

## 红旗 — 停，走流程

如果发现自己在想：
- "先快速修一下，之后调查"
- "改了X试试看行不行"
- "改多个地方一起跑测试"
- "跳过测试，我手动验证"
- "我不完全理解，但这个可能行"
- 还没跟踪数据流就提议解决方案

**这些都意味着：停。回第一阶段。**
