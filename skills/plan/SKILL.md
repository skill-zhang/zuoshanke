# Plan 模式

## 核心理念

在动手实现之前，先写完整的 markdown 计划文档。不执行任何代码，不修改任何文件（计划本身除外）。

**行动前先思考。先计划，再行动。**

## 流程

### 1. 理解需求

- 仔细理解用户的需求
- 澄清模糊之处
- 确定范围和边界

### 2. 设计方案

- 架构选型
- 组件划分
- 数据流设计
- 接口设计

### 3. 写计划文档

在 `.hermes/plans/` 或 `docs/plans/` 下创建 markdown 计划文件：

```markdown
# [功能名称] 设计方案

## 背景
为什么需要这个功能

## 设计目标
- 功能目标
- 非功能目标（性能、安全、可维护性）

## 架构方案
[文字描述 + 可选 ASCII 图表]

## 组件划分
### 组件 1：XXX
- 职责：
- 接口：
- 文件位置：

### 组件 2：YYY
...

## 数据流
[输入 → 处理 → 输出的完整链路]

## 接口设计
[API endpoint、函数签名等]

## 存储设计
[DB schema、文件格式等]

## 实现步骤
1. 步骤一
2. 步骤二
...

## 风险和缓解措施
- 风险：... → 缓解：...
```

### 4. 呈现给用户

展示计划文档的摘要和关键决策点，让用户审阅并点头。

### 5. 根据反馈修订

用户反馈后更新计划文档。循环直到达成一致。

## 规则

- **不执行代码**
- **不修改文件**（除了计划文档本身）
- **不实现功能**
- 聚焦在思考和分析上
- 用搜索工具调研现有代码结构
- 用读文件工具了解现有实现

## 输出

- 计划文档保存到 `docs/plans/` 或项目约定位置
- 版本号递增（v1.0 → v1.1）
- 旧版本保留，不覆盖
