# 测试驱动开发 (TDD)

## 核心理念

先写测试，看它失败，再写最少代码让它通过。

**铁律：没有先写失败测试的生产代码，不算TDD。**

## 何时使用

**始终使用：**
- 新功能开发
- Bug 修复
- 重构
- 行为变更

**例外（需先问用户）：**
- 一次性原型
- 自动生成的代码
- 配置文件

## 红绿重构循环

### 🔴 RED — 写失败的测试

写一个最小测试，描述期望的行为：

```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'
    result = retry_operation(operation)
    assert result == 'success'
    assert attempts == 3
```

**要求：**
- 一个测试只验证一个行为
- 命名清晰（名字里出现"和"？拆开）
- 测试真实代码，不是 mock（除非真的不可避免）
- 描述的是行为，不是实现

### ✅ 验证 RED — 看它失败

**不可跳过。** 运行测试确认：
- 测试失败（而非拼写错误导致的报错）
- 失败信息符合预期
- 失败的原因是功能缺失

```bash
pytest tests/test_feature.py::test_specific_behavior -v
```

**测试直接通过了？** 你在测试已有行为。修改测试。
**测试报错了？** 修复报错，重新运行直到正确失败。

### 🟢 GREEN — 最少代码

写最简单的代码让它通过，不做任何多余的事情：

```python
def add(a, b):
    return a + b  # 不多不少
```

**GREEN 阶段可以作弊：**
- 硬编码返回值
- 复制粘贴
- 重复代码
- 跳过边界情况

### ✅ 验证 GREEN — 看它通过

```bash
# 运行单个测试
pytest tests/test_feature.py::test_specific_behavior -v

# 运行全部测试检查回归
pytest tests/ -q
```

### 🔵 REFACTOR — 清理

通过后才可以重构：
- 消除重复
- 改进命名
- 提取辅助函数
- 简化表达式

保持测试保持通过。不要添加新行为。

### 重复

一个行为一个测试。一次一个循环。

## 为什么顺序重要

**"先写代码，再补测试验证"**

事后补的测试一定通过。通过意味着什么也证明不了：
- 可能测的是错误的东西
- 可能测的是实现，不是行为
- 可能遗漏了遗忘的边界情况
- 你从没见过它抓到 bug

测试先行使你亲眼看到失败，证明它确实在测某样东西。

## 常见借口

| 借口 | 真相 |
|------|------|
| "太简单了不用测" | 简单代码也会出问题。写测试30秒。 |
| "我之后再测" | 事后通过的测试什么都证明不了。 |
| "已经手动测过了" | 手动≠系统化。没有记录，不能重跑。 |
| "删掉X小时的工作太浪费" | 沉没成本谬误。保留不可信赖的代码才是浪费。 |
| "TDD太教条了，务实一点" | TDD本身就是务实。早发现bug比事后调试快。 |

## 验证清单

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试在实现前都确认过失败
- [ ] 每个测试因正确原因失败（功能缺失，不是拼写错误）
- [ ] 写了最少代码让它通过
- [ ] 所有测试通过
- [ ] 运行输出干净（无错误、警告）
- [ ] 边界情况和错误情况已覆盖

## 铁律

```
生产代码 → 测试必须存在且先失败
否则 → 不是 TDD
```
