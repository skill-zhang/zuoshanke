# 写实现计划

## 核心理念

一份好的计划让实现变得显然。如果还需要猜测，计划就不完整。

**每个任务 = 2-5 分钟的小块工作。**

## 何时使用

**始终在以下场景前使用：**
- 实现多步骤功能
- 分解复杂需求
- 委托子任务

**不要跳过，即使：**
- 功能看起来简单（假设导致 bug）
- 你打算自己实现（未来的你需要指导）

## 任务粒度

**太大：**
```markdown
### 任务 1: 构建鉴权系统
[5个文件50行代码]
```

**合适：**
```markdown
### 任务 1: 创建含 email 字段的 User 模型
[10行, 1个文件]

### 任务 2: 给 User 添加密码哈希字段
[8行, 1个文件]
```

## 计划文档结构

### 标题（必需）

```markdown
# [功能名称] 实现计划

**目标：** [一句话描述做什么]

**架构：** [2-3句话说明方法]

**技术栈：** [关键技术/库]
```

### 任务结构

每个任务包含：

```markdown
### 任务 N: [描述性名称]

**目标：** 这个任务做什么（一句话）

**文件：**
- 创建: `exact/path/to/new_file.py`
- 修改: `exact/path/to/existing.py:45-67`
- 测试: `tests/path/to/test_file.py`

**第一步：写失败测试**
[完整代码]

**第二步：运行确认失败**
[命令 + 预期输出]

**第三步：写最少实现**
[完整代码]

**第四步：运行确认通过**
[命令 + 预期输出]

**第五步：提交**
[git 命令]
```

## 原则

### DRY（不要重复自己）
**不好：** 在3个地方复制验证逻辑
**好：** 提取验证函数，全局复用

### YAGNI（你不会需要它的）
**不好：** 为"未来需求"添加灵活性
**好：** 只实现现在需要的

### TDD（测试驱动开发）
每个产生代码的任务都包含完整 TDD 循环。

### 频繁提交
每个任务完成后提交。

## 常见错误

- **模糊任务** — "添加鉴权" → "创建User模型"
- **代码不完整** — 要提供可复制粘贴的完整代码
- **缺少验证** — 必须要"运行这个命令，预期N个通过"
- **缺少文件路径** — 不能"创建模型文件"，要"创建 `src/models/user.py`"

## 记住

```
小块任务（每个2-5分钟）
精确文件路径
完整代码（可复制粘贴）
精确命令 + 预期输出
验证步骤
DRY, YAGNI, TDD
频繁提交
```
