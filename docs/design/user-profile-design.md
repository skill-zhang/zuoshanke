# 坐山客用户画像（User Profile）设计方案

> 让坐山客本体和分身都知道用户的习惯、偏好和工作方式
> 2026-05-26

## 一、Hermes 的做法（参考，不照搬）

Hermes 的 `memory(target='user')` 本质就是一段段描述用户的文字，每次对话自动加载到 prompt 顶部：

```json
[
  "核心工作方式：让坐山客自主开发自己，反对手写代码",
  "验收必须实机测试拿数据说话",
  "偏好AI原生设计、极简表单、干燥CSS",
  "UI细节极度敏感：按钮同类一致、px级定位",
  "调试纪律：报错先ls再session_search",
  ...
]
```

| 特性 | Hermes 的实现 |
|------|-------------|
| 存储 | 内存 + 序列化持久化 |
| 每条格式 | 纯文本字符串 |
| 注入方式 | 每次对话自动拼到 system prompt |
| 写入方式 | `memory(action='add', target='user')` |
| 更新方式 | 手动 `replace` / `remove` |
| 大小限制 | 1,375 字符 |
| 过期机制 | 无 |

坐山客不能直接复用这个机制，但目标一致：**让系统知道跟谁在对话、那人喜欢什么**。

## 二、坐山客的实现

### 2.1 存储

利用已有的 `AgentMemory` 表，加一个 `category='user'` 标记用户画像条目：

```python
AgentMemory(
    category="user",       # 跟 category="agent"（设计哲学）区分
    scope="zhu",           # 本体级，所有场景可读
    key="work_style",      # 唯一标识，方便查找和修正
    content="核心工作方式：让坐山客自主开发自己，反对手写代码",
    tags=["工作流", "铁律"],
    base_weight=9,         # 重要性 1-10
    priority_level="P0",   # P0/P1/P2/P3
    is_immortal=True,      # P0/P1 不朽不衰减
    source="user",         # 来源：user / llm / auto
)
```

不需要新表，`create_all()` 零破坏。

### 2.2 优先级分层

| 等级 | 意义 | base_weight | is_immortal | 示例 |
|------|------|------------|------------|------|
| **P0** | 铁律/原则 | 8-10 | ✅ | 「先讨论再动手」「只信拨测不信CSS」 |
| **P1** | 偏好/习惯 | 6-7 | ✅ | 「偏好极简表单」「温度0.7」 |
| **P2** | 参考信息 | 4-5 | ❌ | 「喜欢侧栏折叠」「暗色主题」 |
| **P3** | 临时/上下文 | 2-3 | ❌ | 「今天要优先查GPU占用」 |

P0 和 P1 永远不被淘汰，P2/P3 当容量不足时按最后访问时间淘汰。

### 2.3 注入方式

修改 `context_builder.py` 的 `_build_memory_block()`，在现有记忆注入后追加用户画像段落：

```
─────────────────────────────────
👤 用户画像（User Profile）
你正在跟一个这样的用户对话：
- 【P0】核心工作方式：让坐山客自主开发自己，反对手写代码
- 【P0】验收必须实机测试拿数据说话
- 【P1】偏好极简表单、干燥CSS方案
- 【P1】UI细节极度敏感：按钮同类一致、px级定位
─────────────────────────────────
```

**注入规则：**
- P0 + P1：**始终注入**所有条目（上限 ~8 条，约 300 字符）
- P2：按话题匹配注入（场景 context 中有相关关键词时触发）
- P3：不自动注入，仅当用户主动提及相关话题时查

**何时注入：** 所有场景/channel 的初始 context 都注入。本体（闲聊频道/秘密花园）全量注入。

### 2.4 写入通道

三个通道写入用户画像：

| 通道 | 写入者 | 最高可写等级 | 说明 |
|------|--------|------------|------|
| **工具调用** | 任意场景的 LLM | P1 | 分身聊天中发现用户偏好 → 调 `user_profile_add` 写入 |
| **闲聊频道** | 本体（坐山客） | P0 | 本体可直接写入任何等级 |
| **SettingsView** | 用户自己 | P0 | 前端手动增删改 |

### 2.5 容量管理

利用 `AgentMemory` 已有的 `prune()` 方法，专门对 `category='user'` 执行：

```
触发条件：user 条目 > 30 条 或 总字符 > 1,500
执行策略：
  1. P0 + P1 跳过（is_immortal=True）
  2. P2 按 last_accessed_at 排序，淘汰最早 20%
  3. P3 全部淘汰，保留最后 2 条
  4. 合并 tags 相同的内容相近条目（相似度 > 70%）
```

触发时机：`idle_extractor.py` 的 60 秒扫描循环中顺带执行。

## 三、边界情况

### 3.1 分身写了错误的偏好

分身误判用户偏好，写了错误内容 →
1. 用户在聊天中纠正 → 分身调 `user_profile_correct()` → 旧值归档到 correction_trail，权重 +2
2. 用户直接在 SettingsView 里改

### 3.2 用户变了偏好

之前喜欢 A，现在喜欢 B →
1. 新条目 `base_weight + 2`（因为是最新表达的，更可信）
2. 旧条目不删除，加标签 `superseded_by=<新key>`
3. context 注入时只取最新的那条

### 3.3 多用户场景

坐山客目前是单用户系统（本体对话就是跟张清泉）。如果以后支持多用户：
- `AgentMemory` 表加一个 `user_id` 字段即可隔离
- 每个用户登录后只加载自己的 user profile
- 当前不需要，schema 预留

## 四、验证方法

```bash
# 1. 分身写入
curl -X POST http://localhost:8000/api/scenes/xxx/stream \
  -d '{"content":"我习惯先看数据再说话"}'
# → 分身 LLM 识别到用户偏好 → 调 user_profile_add()
# → 检查 agent_memory 表: category='user', scope='zhu', content='先看数据再说话'

# 2. 查询
curl http://localhost:8000/api/user-profile
# → {"P0": [{content, tags, key}], "P1": [...], "P2": [...], "total": N}

# 3. context 注入验证
# 进入任意场景 → 查看构建 prompt 底部是否有「👤 用户画像」段落

# 4. 容量修剪
# 手动注入 35 条 P3 → 触发 prune → 检查剩余条目数 ≤ 30
```

```
  5. 控制注入字符数

  6. SettingsView 加第 5 Tab「用户画像」
  7. API: GET/POST/PUT/DELETE /api/user-profile

Phase 4（自动维护 → 不用手动清理）:
  8. idle_extractor.py 加入 prune 逻辑
  9. 容量报警
```
