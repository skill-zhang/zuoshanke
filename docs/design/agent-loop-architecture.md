# Agent Loop 架构设计

> 坐山客 AI 工作台 · 收敛-排序-聚焦-反馈循环
> v1.0 · 2026-05-18

---

## 一、问题陈述

当前坐山客的 Thinking Map（思维图）只有**发散（Divergence）** 能力——当用户提出一个复杂问题，TM 会不断分裂出子节点、子子节点，越来越散。这本质上是头脑风暴，本身是好的，但：

| 缺失阶段 | 后果 |
|---------|------|
| ❌ **收敛（Converge）** | 节点越分越多，没有合并去重，没有关闭不可行分支 |
| ❌ **排序优先级（Prioritize）** | 所有节点平铺，看不出先后顺序和依赖关系 |
| ❌ **聚焦执行（Focus Execute）** | Action Map 展示所有节点，分散注意力，没有 WIP 限制 |
| ❌ **反馈调整（Reflect）** | 执行完的结果不会反哺回 Thinking Map，循环是死的 |

### 类比：我们做系统本身的过程

回想坐山客开发过程：
1. 先定三层架构（场景广场/工坊/频道）→ 再做细节
2. 先做后端 API → 再做前端组件 → 再做集成测试
3. 不是所有方向同时开干——因为有依赖关系和优先级

Thinking Map 现在缺的就是这个**依赖分析和优先级排序**的能力。

## 二、完整 Agent Loop

```
                    ┌───────────────────────────┐
                    │        Thinking Map        │  ← 发散（已有）
                    │    (Divergence / Explore)   │     头脑风暴、拆解子问题
                    │      自由分裂节点            │
                    └────────────┬───────────────┘
                                 │
                                 ▼
                    ┌───────────────────────────┐
                    │         Converge           │  ← 收敛（新增）
                    │   (Cluster + Reduce)       │     聚类去重、合并相似项
                    │     关闭不可行分支           │     标记已知已存在方案
                    └────────────┬───────────────┘
                                 │
                                 ▼
                    ┌───────────────────────────┐
                    │       Prioritize Sort      │  ← 排序（新增）
                    │   (DAG + Priority Queue)   │     依赖分析（拓扑排序）
                    │     区分高/中/低优先级      │     确定并行/串行
                    └────────────┬───────────────┘
                                 │
                                 ▼
                    ┌───────────────────────────┐
                    │      Focus & Execute       │  ← 聚焦执行（改进）
                    │   (Action Map · WIP 限制)   │     Action Map 只展示顶部1-3项
                    │       一次聚焦一件事         │     执行完一个取下一个
                    └────────────┬───────────────┘
                                 │
                                 ▼
                    ┌───────────────────────────┐
                    │     Reflect & Feedback     │  ← 反馈调整（新增）
                    │   (结果反哺 Thinking Map)   │     "这条路不通"→标记 closed
                    │      新发现→新增子节点       │     重新触发收敛+排序
                    └────────────┬───────────────┘
                                 │
                                 └─────────→ 回到发散（如需继续）或 结束
```

### 各阶段详解

#### 阶段 1：发散（Divergence）— 已有，不动

Thinking Map 当前的自由分裂机制保持不变。用户提出目标后，系统自动生成子节点。

#### 阶段 2：收敛（Converge）— 新增

**目的**：把发散的结果做结构化整理。

**算法流程**：
1. **节点聚类**：用语义相似度（或 LLM 辅助）将相似节点归为一类
   - 例："写登录页"、"做登录表单"、"登录功能" → 聚类为"用户登录功能"
2. **去重合并**：同一类内的节点合并为一个代表性节点
3. **不可行分支关闭**：明显不可行或已被用户否定的分支 → 标记 `status: discarded`
4. **已知方案标记**：已有现成工具/技能/方案的分支 → 标记 `has_existing_solution: true`

**输出**：一个收敛后的候选任务列表。

#### 阶段 3：排序优先级（Prioritize）— 新增

**目的**：确定先做什么后做什么，哪些可以并行。

**算法流程**：
1. **依赖分析（DAG 构建）**：分析任务间的依赖关系
   - "先有 A 才能做 B" → A → B 有向边
   - "A 和 B 互不依赖" → 可并行
2. **拓扑排序**：生成一个满足所有依赖的执行顺序
3. **优先级标记**：LLM 或用户辅助标记高/中/低
4. **生成执行队列**：一个有序的线性列表

**UI 交互**：
- 用户可以拖拽调整顺序
- 用户可以手动标记依赖关系
- 可以强行指定 "这个先做"

**输出**：一个有序的 Priority Queue。

```
Priority Queue 示例（为"做一个天气推荐系统"）：
┌── 1. [高] 调研天气API（无依赖）
├── 2. [高] 设计推荐算法（依赖 1）
├── 3. [中] 搭建Web界面（依赖 1）
├── 4. [中] 用户反馈收集（依赖 3）
└── 5. [低] 多语言支持（依赖 3、4，可后置）
```

#### 阶段 4：聚焦执行（Focus Execute）— 改进 Action Map

**目的**：一次只聚焦一件事（或一组并行不冲突的事），避免认知过载。

**改动点**：
- Action Map **不再展示所有节点**
- 只从 Priority Queue 顶部取 **1-3 个**节点展示（WIP 限制）
- 执行完一个，自动从队列取下一个
- 已完成节点折叠收起，用户可展开查看历史
- 未开始的任务不在 Action Map 中显示

**UI 交互**：
- Action Map 左侧显示当前聚焦区域，右侧显示队列待办
- 用户可手动调整 WIP 限制（1/2/3）
- 用户可强制 "跳过当前，先做后面的"

#### 阶段 5：反馈调整（Reflect）— 新增

**目的**：让执行结果反哺回 Thinking Map，形成活循环。

**流程**：
1. 执行完成一个节点后，将结果摘要注入 Thinking Map
2. 如果"这条路不通" → 标记 `status: blocked` 或 `failed`
3. 如果执行中发现新的子问题 → 以新子节点挂回 Thinking Map
4. 如果用户中途改变主意 → 手动调整队列
5. **重新触发收敛+排序**：队列可能因此变化

**输出**：
- 更新后的 Priority Queue
- 更新后的 Thinking Map（含已执行节点的结果）

## 三、状态机与节点生命周期

```
                        ┌──────────┐
                        │  created  │  ← 发散阶段生成
                        └────┬─────┘
                             │
                             ▼
                        ┌──────────┐
                        │  refined  │  ← 收敛后，表示已去重合并
                        └────┬─────┘
                             │
                             ▼
                     ┌───────────────┐
                     │  prioritized  │  ← 已排入 Priority Queue
                     └───────┬───────┘
                             │
                             ▼
                     ┌───────────────┐
                     │   in_queue    │  ← 在队列中等待执行
                     └───────┬───────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌──────────┐  ┌───────────┐  ┌──────────┐
        │  active  │  │  blocked  │  │ skipped  │
        └────┬─────┘  └───────────┘  └──────────┘
             │              │              │
             ▼              │              │
        ┌──────────┐        │              │
        │ completed│◄───────┘              │
        └──────────┘                       │
                                           ▼
                                     ┌──────────┐
                                     │ discarded│
                                     └──────────┘
```

### 状态定义

| 状态 | 含义 | 可见性 |
|------|------|--------|
| `created` | TM 刚分裂出来 | TM 上可见 |
| `refined` | 已被收敛合并 | TM 上可见（合并后） |
| `prioritized` | 已分配优先级 | 队列中可见 |
| `in_queue` | 在 Priority Queue 中等待 | 队列中可见 |
| `active` | 正在 Action Map 中执行（WIP） | AM 上可见 |
| `blocked` | 依赖项未满足或被外部阻断 | 队列中标记 |
| `skipped` | 用户要求跳过 | 折叠状态 |
| `completed` | 执行完成，有结果摘要 | 折叠可展开 |
| `discarded` | 不可行分支，已关闭 | 折叠可查看（为什么关闭） |

## 四、设计原则与模式溯源

| 概念 | 来源 | 对应我们的 |
|------|------|-----------|
| **收集→整理→组织→回顾→执行** | GTD | Thinking Map = 收集/整理，Converge = 组织，Prioritize = 回顾 |
| **可视化工作流、限制WIP** | Kanban | Focus Execute 的 WIP 限制，队列就是看板 |
| **计划→执行→检查→改进** | PDCA | Reflect & Feedback 闭环 |
| **有依赖关系的任务排序** | 拓扑排序（DSA） | Prioritize Sort 的 DAG |
| **发散→收敛→再发散→再收敛** | 双钻模型 | 外层循环也是这个模式 |
| **一次只做一件事** | 单任务聚焦 | WIP=1 的本质 |
| **认知负荷理论** | 心理学 | 同时展示过多节点会降低效率 |

我们不是发明新东西，是把这些成熟的模式**嵌入到 AI Agent 的自循环中**，让 AI 不只是一次性问答，而是像一个真正的开发者一样结构化地推进工作。

## 五、与现有系统的关系

### 与现有 API 的关系

```
Thinking Map API → 增加阶段控制参数
  GET  /api/thinking-map/{id}          # 获取 TM（含状态）
  POST /api/thinking-map/{id}/converge # 触发收敛
  POST /api/thinking-map/{id}/sort     # 触发排序

Priority Queue API → 新增
  GET  /api/priority-queue/{session_id} # 获取队列
  PUT  /api/priority-queue/{session_id}/reorder   # 拖拽排序
  POST /api/priority-queue/{session_id}/next      # 取下一个执行项
  PUT  /api/priority-queue/{session_id}/wip       # 设置 WIP 限制

Action Map API → 调整
  原有 execute 端点不变
  新增 /api/action-map/{session_id}/focus  # 获取当前聚焦的节点
  新增 /api/action-map/{node_id}/feedback  # 执行反馈注入
```

### 与 SSE 协议的关系

新增 SSE 事件：
| 事件 | 触发时机 | 数据字段 |
|------|---------|----------|
| `converge_start` | 收敛开始时 | `node_count`, `clusters` |
| `converge_done` | 收敛完成 | `before`, `after`, `merged_pairs` |
| `sort_done` | 排序完成 | `queue`（有序列表）|
| `focus_change` | Action Map 聚焦任务变化 | `old_node`, `new_node` |
| `reflect_update` | 反馈注入完成 | `node_id`, `result_summary`, `new_subnodes` |

### 与前端组件的关系

新增/改造组件：
- **ConvergePanel** — 收敛结果展示，显示合并前后的变化
- **PriorityQueueView** — 队列展示，可拖拽排序，显示依赖箭头
- **ActionMapFocus** — 改造后的 Action Map，只显示 WIP 节点
- **ReflectTimeline** — 反馈时间线，显示执行结果如何影响 TM

## 六、实现优先级

| 优先级 | 功能 | 依赖 |
|--------|------|------|
| P0 | Converge 阶段 — 聚类去重 + 合并 | 无 |
| P1 | Priority Queue — DAG + 拓扑排序 | 依赖 P0 |
| P1 | Action Map WIP 限制 — 只显示 1-3 个节点 | 无（独立改动）|
| P2 | Reflect & Feedback — 执行结果反哺 TM | 依赖 P0、P1 |
| P2 | UI 交互 — 拖拽排序、依赖编辑 | 依赖 P1 |
| P3 | 自动重触发 — 反馈后自动重新收敛+排序 | 依赖 P2 |

## 七、边界情况与注意事项

### 空队列
- 队列为空时，Action Map 显示 "没有待办任务，返回思维图继续探索？"
- 此时如果 TM 还有 `created` 状态的节点 → 提示用户是否要收敛

### 所有节点被阻塞
- 一个节点被阻塞时，尝试跳过它取下一个
- 如所有节点均阻塞 → 显示 "所有路径均受阻，是否需要调整方向？"

### 用户手动干预
- 用户可随时打断循环，手动拖拽排序、跳过、关闭节点
- 用户可直接在 TM 上新增节点，新增后自动触发收敛

### 超大 TM（100+ 节点）
- 收敛阶段先做粗粒度聚类（K=5-10），再做细粒度合并
- 避免一次性将 100 节点全部展示给 LLM 做排序（token 爆炸）
- 分层排序：先大类排序，再类内排序

### 与现有记忆系统的关系
- Reflect 阶段的反馈结果也应存入记忆系统（`Memory Manager`），作为上下文供后续对话使用
- 标记 "已完成的执行路径" 避免后续重复发散到同一方向

## 八、附录：关键术语

| 术语 | 定义 |
|------|------|
| **Thinking Map (TM)** | 思维图，用于发散探索、拆解问题 |
| **Action Map (AM)** | 行动图，用于聚焦执行任务 |
| **Converge** | 收敛，聚类去重合并 |
| **Priority Queue** | 优先级队列，有序的任务执行列表 |
| **WIP (Work In Progress)** | 进行中工作限制 |
| **DAG (有向无环图)** | 表示任务依赖关系的数据结构 |
| **拓扑排序** | 有依赖关系的任务排序算法 |
| **Reflect / Feedback** | 反馈调整，执行结果反哺思维图 |
| **Node 生命周期** | created → refined → prioritized → in_queue → active → completed |
| **双钻模型** | 发散→收敛→再发散→再收敛的设计思维模型 |
| **Agent Loop** | 智能体持续执行直到达成目标的循环 |
