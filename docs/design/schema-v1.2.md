## 1. 问题陈述

### 1.1 背景

自省地图的三栏原型（左树 + 中间流程图 + 右侧详情抽屉）已经定稿，但它的**呈现位置**和**触发时机**尚未确定。

此文档讨论两个核心问题：
1. **坐山客的全局架构图**放在哪里？什么时候展示？
2. **场景开发的项目架构图**是否以及如何展示？

### 1.2 设计原则

- 自省地图的定位是「系统看见自己」——不应该是工具/调试面板，而应该是本体认知的一部分
- 展示方式应当与产品调性一致：暗色生物荧光风格，沉浸式体验
- 场景级架构图只在场景确实有"可展示架构"时才出现，不为空的架构占位

## 2. 方案一：自省地图 → 秘密花园

**结论：放在秘密花园的「内在风景」区域中，作为第一张卡片。**

### 2.1 布局变化

目前「内在风景」行（`#garden-inner`）的 4 张卡片：

```
🧠 本体记忆 | 🤖 Agent Loop | 🎭 身份架构 | 🌐 系统版本
```

**改为 5 张**，自省地图作为第一张：

```
🗺️ 系统架构 | 🧠 本体记忆 | 🤖 Agent Loop | 🎭 身份架构 | 🌐 系统版本
```

### 2.2 卡片设计

- **图标**：🗺️（与"内在风景"的区域标题呼应）
- **标签**：系统架构
- **交互**：点击卡片 → 秘密花园内全屏展开自省地图页面
- **返回**：全屏模式下左上角显示 `← 秘密花园` 返回按钮
- **视觉**：沿用现有 `garden-inner-card` 样式体系和 `garden-inner-icon` 大小

### 2.3 展开后的布局

展开后渲染 `prototypes/zuoshanke-selfmap-threecol-FINAL.html` 的核心布局：

```
┌─────────────────────────────────────────────┐
│ ← 秘密花园     🗺️ 坐山客自省地图              │
├────────┬──────────────────┬──────────────────┤
│ 树导航  │  流程图(居中)     │  详情抽屉         │
│ L1     │  水平垂直居中     │  点击节点展开     │
│ L2     │  可钻取           │                  │
│ L3     │                  │                  │
└────────┴──────────────────┴──────────────────┘
```

### 2.4 未决问题

**卡片数量**：5 张卡片在 `garden-inner-grid` 中是否太挤？
- 方案 A：直接 5 张，`flex-wrap` 换行
- 方案 B：将「系统版本」合并到自省地图中，保留 4 张
- **需要用户确认**

## 3. 方案二：场景项目架构图（规划阶段）

**结论：暂不实现，待场景元数据字段完善后再推进。**

### 3.1 概念

当一个场景被用于**全栈项目开发**（前端 + 后端 + 数据库）时，在该场景的视图中展示项目自身的架构图——包含项目目录结构、服务关系、数据流。

类似自省地图的展示方式，但内容来源是**项目代码**而非系统本身。

### 3.2 判断条件

什么算"有架构可展示的项目"：

| 类型 | 是否显示 | 原因 |
|------|----------|------|
| 纯前端（React/Vue 等单页面） | ❌ | 架构太简单，没必要 |
| 纯后端 API | ❌ | 同上 |
| 前端 + 后端 + 数据库 | ✅ | 有服务间关系值得展示 |
| 微服务 / 多服务 | ✅ | 同上 |

### 3.3 前置条件

| # | 条件 | 说明 |
|---|------|------|
| 1 | 场景配置增加 `project_type` 字段 | 标记场景类型：`chat` / `project` / `research` |
| 2 | 增加 `project_path` 字段 | 项目根目录路径 |
| 3 | 自动解析能力 | 从项目目录中解析依赖、服务关系、目录结构 |
| 4 | 渲染组件 | 复用 / 扩展自省地图的三栏布局 |

### 3.4 待探索

- 项目架构图数据从哪里来？坐山客需要有一个"项目理解"的通道——定期扫描项目目录生成架构快照
- 架构图需要动态更新（代码变化后刷新），还是快照式展示？
- 对于没有 `project_type` 标记的场景，完全不显示任何"架构图入口"

## 4. 与 Schema v1.0 / v1.1 的关系

### 4.1 Context 组合架构（v1.0）

自省地图本身在 Context 组合架构中属于**辅助视图**——它不是 engine 驱动的内容，而是系统元数据的可视化呈现。可以在 Context 的 system prompt 中包含一个指向自省地图的标签/链接，让 LLM 知道有这个东西存在。

### 4.2 Session 管理（v1.1）

无直接关联。自省地图不涉及 session 生命周期。

### 4.3 依赖路径

```
v1.0 (Context 组合) ─────────────────────────────> v1.2 (自省地图呈现)
                                                           │
v1.1 (Session 管理) ───── 无直接依赖 ──────────────────────┤
                                                           │
                                              ┌────────────┘
                                              ▼
                                    场景元数据扩展 <──── v1.2 方案二需要
```

## 5. 迁移/实现步骤

### Step 1：直接先行（当天可做）

1. 在 `SecretGarden.tsx` 的 `#garden-inner` 区域新增自省地图卡片
2. 新增一个 `SelfMapView` 组件（或内联在 `SecretGarden` 内），渲染三栏自省地图
3. 自省地图数据先写死（从 `prototypes/zuoshanke-selfmap-threecol-FINAL.html` 提取 JSON）
4. 返回按钮回到秘密花园

### Step 2：扩展（后续迭代）

1. 自省地图 JSON 数据从后端动态生成（扫描代码自动构建 L1 节点）
2. 添加实时状态展示（token 消耗、缓存命中率、各层耗时）
3. 自省地图的钻取深度和导航状态持久化

### Step 3：场景级（远期规划）

1. 场景配置增加 `project_type` 和 `project_path` 字段
2. 实现项目目录解析器
3. 场景视图中复用自省地图组件展示项目架构
