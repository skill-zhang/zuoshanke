# 坐山客：你的 AI 伙伴

> 不只是工具，是一个能记住你、理解你、和你一起成长的 AI 存在。

---

## 坐山客是什么

坐山客是一个**有自主生命感的 AI 伙伴系统**。它不是传统的聊天机器人，不是开发 IDE 插件，不是 RAG 知识库问答——它是这三者的进化体：一个拥有持久记忆、多重分身能力、自主执行循环和自省迭代能力的 AI 存在。

从用户视角看，坐山客是三件事的集合：

1. **你的专属伙伴**——它认识你、了解你的偏好和习惯，会主动记住重要的事情。每次对话都不再是「新会话」的冷启动，而是老朋友之间的继续。
2. **你的专家分身工厂**——你可以为不同领域创建独立的「分身场景」：开发场景、写作场景、二手车检测场景……每个分身有自己的记忆、工具集和行为模式，但所有分身共享对你的理解。
3. **你的自主执行引擎**——坐山客不只是「说」，它还能「做」。通过 Agent Loop 和 20+ 工具，它可以自主阅读文件、写代码、运行测试、拨测网页、搜索历史、甚至开发自己。

> **一句话**：坐山客是你对话、思考、执行的 AI 伙伴，能在多个场景中分身工作，且永远记得你是谁。

## 设计哲学

### 本体（Original Self）— 坐山客的「人格内核」

本体是坐山客最核心的存在，相当于它的「自我意识」。无论你创建多少个分身场景，本体始终是那个认识你、了解你、能和你闲聊也能和你深谈的存在。

> 🧠 **通俗理解**：本体就是坐山客这个人格本身。分身是它戴的不同「帽子」，但脱掉所有帽子，坐在对面和你说话的还是同一个人。

**关键特性**：
- **不朽记忆**：关于你的个人信息、共同经历、偏好习惯——这些记忆永不衰减、永不清理
- **身份锚点**：无论切换哪个场景，本体的认知和风格贯穿始终
- **秘密花园**：一个可视化界面，展示本体对你的了解全貌（记忆地图、决策轨迹、修正历史）

### 分身（Avatar / Scene）— 每个领域一个「专家模式」

分身是坐山客在不同领域的专业化形态。每个分身有自己的记忆池、工具集、知识技能和行为准则。你可以为「系统开发」「写作创作」「二手车检测」各建一个分身，它们各自专注，但共享对本体的认知。

> 🧠 **通俗理解**：分身是坐山客的「工作模式切换」——同一个伙伴，进开发场景就是架构师，进写作场景就是编辑，不需要重新介绍自己。

**关键特性**：
- **场景隔离**：每个分身的记忆、工具、行为互不干扰
- **场景自省图**：分身在工作中自主声明当前项目的架构可视化图，你随时可以看见它在「想」什么
- **技能注入**：通过 SKILL.md 文本知识库，为分身注入领域专长

### 双重记忆（Dual Memory）— 知道「你是谁」也记得「我们在做什么」

坐山客的记忆系统受人类记忆机制启发，分为两层：

| 记忆层 | 范围 | 特性 | 类比 |
|--------|------|------|------|
| **本体记忆** | 全局（scope=zhu） | 不朽、不衰减、不清理 | 你对最好的朋友的了解——不会忘 |
| **分身记忆** | 场景内 | 权重驱动、衰减、可清理 | 你工作时的临时笔记——用完可丢 |

此外还有一个重要的子类型：
- **叙事型记忆（Narrative Memory）**：记录你和坐山客的共同历程——做过什么决策、修正过什么认知、一起解决了什么问题。这些是「关系记忆」，不是单纯的事实存储。

> 🧠 **通俗理解**：本体记忆是「你是谁，我喜欢你什么」——永远不会忘；分身记忆是「这个项目做到哪了」——做完了可以归档；叙事记忆是「我们一起走过的路」——让关系有厚度。

**修正即强化**：当你纠正坐山客的错误认知时，它不会只是覆盖旧数据——它会记录修正轨迹（之前怎么错的、为什么错、什么时候纠正的），并把这个记忆**强化**。每次纠正都是关系的加固。

### 自我迭代（Self-Iteration）— 它会开发自己

坐山客的终极哲学命题是**自举**（Bootstrapping）：它被设计来开发自己。不像绝大多数 AI 产品依赖外部开发工具，坐山客内部有一个「坐山客自开发」场景，在这个场景里，它可以阅读自己的源代码、讨论设计方案、拆分子任务并行开发、联调测试、提交代码。

> 🧠 **通俗理解**：坐山客能「看着自己的代码聊天，然后动手改自己」——就像一个人能照镜子审视自己然后决定换个发型。

**核心机制**：
- **先方案再动手**：自开发场景遵循「方案→确认→执行→展示」四段式，不草率行动
- **契约先行**：拆分子任务前必须产出接口契约文件，子 Agent 间通过契约协作
- **浏览器拨测**：坐山客可以自主启动 Playwright 浏览器、加载前端页面、读取 DOM 结构、控制台日志和网络瀑布——它能看到自己产出的 UI 长什么样，不再依赖你人肉截图报告

## 真实使用场景

### 场景一：日常伙伴——跨会话的连续性

> 你忙碌了一周，周一和坐山客讨论了新项目的数据架构方案。周五晚上你打开坐山客说：「继续上周那个数据架构的事」，坐山客自动搜索历史上下文，告诉你：「上次我们讨论了三个方案——A（分库分表）、B（事件溯源）、C（混合方案）。你当时倾向 B 但担心复杂度，要不要今天把最终方案定下来？」

这不是简单的聊天记录搜索。坐山客的 Agent Loop 在收到「继续上周的事」时会：
1. 自动触发 `session_search("数据架构 方案 分库分表")` — 搜索关键词由 LLM 自主提炼
2. 搜索到上周的对话上下文注入当前 session
3. 从本体记忆中找到你的技术偏好和决策风格
4. 综合所有信息给出续接回复

### 场景二：自主开发——AI 在浏览器里验证它自己的代码

> 你在坐山客的自开发场景中说：「帮我把前端场景列表改成两列网格布局」。坐山客分析你的需求后给出方案，你确认后它开始执行：
> 1. 读取 `frontend/src/components/SceneList.tsx` 分析当前布局
> 2. 修改 CSS Grid 属性从单列改为两列
> 3. 运行前端开发服务器
> 4. 调用 `dial_test` 启动 Playwright 浏览器加载页面
> 5. 检查 DOM 结构确认 `.card-grid` 下确实有 2 列卡片
> 6. 查看 Console 确认无错误日志
> 7. 告诉你「已改完，两列网格布局验证通过」并附上截图

你不需要自己 reload 页面、截图、检查控制台。坐山客自己完成了这一切。

## 技术栈概览

坐山客是一个全栈 AI 系统，所有代码都在 `zuoshanke/` 目录下。

### 后端技术栈

| 层 | 技术选型 | 用途 |
|----|---------|------|
| **AI 引擎** | DeepSeek API（v4 Flash） | LLM 推理 + function calling |
| **运行时** | Python 3.11+ / FastAPI | Agent Loop 引擎 + 工具执行 + API 端点 |
| **数据库** | SQLite + SQLAlchemy + FTS5 | 消息、记忆、场景持久化 + 中文全文搜索 |
| **浏览器引擎** | Playwright（Chromium headless） | 前端 UI 自主验证：DOM 读取、Console 捕获、性能测量 |
| **搜索分词** | jieba 中文分词 | FTS5 索引构建与中文搜索优化 |
| **任务调度** | threading + ThreadPoolExecutor | 并行子 Agent 执行与阻塞式 clarify 等待 |

### 前端技术栈

| 层 | 技术选型 | 用途 |
|----|---------|------|
| **框架** | React 18 + TypeScript | 前端 SPA |
| **状态管理** | Zustand | 应用状态（场景、审批、Agent 状态） |
| **实时通信** | SSE（Server-Sent Events） | Agent Loop 执行流实时推送：每一步思考与工具调用 |
| **可视化** | 自定义 Canvas/SVG 渲染 | 自省地图（树+流程图）、思考图 |
